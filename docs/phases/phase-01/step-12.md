# Step 12 — Database Schema
### File: `internal/store/migrations/001_initial_schema.up.sql`

---

## What does this file do?

This SQL file **creates all the tables** Orion needs in PostgreSQL. It is applied using `golang-migrate` — a tool that tracks which SQL files have been run and applies new ones in order.

---

## Why migrations instead of writing SQL manually?

Without migrations:
- You manually run SQL on your machine
- A teammate manually runs different SQL on theirs
- Production gets changes in a different order
- Everything breaks

With migrations, every change is a numbered file:
```
001_initial_schema.up.sql    ← this file (create everything)
002_add_tags_column.up.sql   ← future change
003_new_index.up.sql         ← future change
```

`golang-migrate` tracks which files ran in a `schema_migrations` table. Anyone runs `make migrate-up` and gets the exact same database state.

## The 5 tables

### Table 1: `jobs` — the central table

```sql
CREATE TABLE jobs (
    id               UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    idempotency_key  VARCHAR(255) UNIQUE,
    name             VARCHAR(255) NOT NULL,
    type             VARCHAR(100) NOT NULL,    -- 'inline' or 'k8s_job'
    queue_name       VARCHAR(100) NOT NULL DEFAULT 'default',
    priority         SMALLINT NOT NULL DEFAULT 5,
    status           VARCHAR(50) NOT NULL DEFAULT 'queued',
    payload          JSONB NOT NULL DEFAULT '{}',
    max_retries      SMALLINT NOT NULL DEFAULT 3,
    attempt          SMALLINT NOT NULL DEFAULT 0,
    worker_id        VARCHAR(255),
    error_message    TEXT,
    -- ... timing columns ...
);
```

**Design decisions explained:**

| Choice | Why |
|---|---|
| UUID primary key | Globally unique without a DB sequence. Can be generated by client before INSERT. Safe in URLs (doesn't reveal record count). |
| JSONB payload | ML job payloads vary wildly between job types. No need to add columns for each new type. |
| SMALLINT priority | Enables `ORDER BY priority DESC` efficiently. CHECK constraint rejects invalid values at DB level. |
| status CHECK constraint | Belt-and-suspenders: DB rejects invalid status even if Go code has a bug. |

## The 3 load-bearing indexes — and why they matter

### Index 1: `idx_jobs_queued_priority`
```sql
CREATE INDEX idx_jobs_queued_priority
    ON jobs (priority DESC, created_at ASC)
    WHERE status = 'queued';
```

**Used by:** Scheduler dispatch loop — runs every 2 seconds.

**What it's for:**
```sql
SELECT * FROM jobs
WHERE status = 'queued'
ORDER BY priority DESC, created_at ASC
LIMIT 50;
```

**Why a partial index (`WHERE status = 'queued'`)?**

A normal index indexes ALL rows. A partial index only indexes rows matching the condition.

Scenario: 1,000,000 total jobs, 50 currently queued.

| Index type | Rows scanned | Time |
|---|---|---|
| No index | 1,000,000 | ~500ms |
| Normal index | 1,000,000 entries | ~10ms |
| Partial index | 50 entries | ~0.02ms |

**20,000x faster.** And the partial index is physically smaller — it fits entirely in RAM.

---

### Index 2: `idx_jobs_retry_eligible`
```sql
CREATE INDEX idx_jobs_retry_eligible
    ON jobs (next_retry_at ASC)
    WHERE status = 'failed' AND next_retry_at IS NOT NULL;
```

**Used by:** Scheduler retry promotion loop.

**Query it serves:**
```sql
SELECT * FROM jobs
WHERE status = 'failed'
AND next_retry_at <= NOW()
AND attempt < max_retries;
```

Only indexes failed jobs with a pending retry time. All other jobs invisible to this index.

---

### Index 3: `idx_jobs_running_worker`
```sql
CREATE INDEX idx_jobs_running_worker
    ON jobs (worker_id, updated_at)
    WHERE status = 'running';
```

**Used by:** Scheduler orphan reclaim sweep — runs every 30 seconds.

**Query it serves:**
```sql
SELECT id FROM jobs
WHERE status = 'running'
AND worker_id NOT IN (
    SELECT id FROM workers WHERE last_heartbeat > NOW() - INTERVAL '90 seconds'
);
```

Only indexes currently running jobs. On a healthy system most jobs are completed — this partial index stays tiny.

## Table 2: `job_executions` — the audit log

```sql
CREATE TABLE job_executions (
    id           UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    job_id       UUID NOT NULL REFERENCES jobs(id) ON DELETE CASCADE,
    attempt      SMALLINT NOT NULL,
    worker_id    VARCHAR(255),
    status       VARCHAR(50) NOT NULL,
    started_at   TIMESTAMPTZ,
    finished_at  TIMESTAMPTZ,
    exit_code    INT,
    logs_ref     TEXT,    -- "s3://orion-logs/jobs/abc/attempt-2.log"
    error        TEXT,
    CONSTRAINT job_executions_unique_attempt UNIQUE (job_id, attempt)
);
```

**Critical rule: NEVER UPDATE rows here. Only INSERT.**

If a job runs 3 times (2 failures + 1 success), there are 3 rows:

| attempt | status | error | started_at | finished_at |
|---|---|---|---|---|
| 1 | failed | OOM killed | 10:00:00 | 10:00:02 |
| 2 | failed | Timeout | 10:05:17 | 10:15:17 |
| 3 | completed | null | 10:35:44 | 10:41:22 |

Full history. Nothing is lost or overwritten.

`logs_ref` stores a pointer (S3/GCS URI) to where actual logs are stored — not the logs themselves. Logs can be gigabytes; PostgreSQL is not a log store.

## Table 3: `workers` — heartbeat registry

```sql
CREATE TABLE workers (
    id             VARCHAR(255) PRIMARY KEY,   -- hostname or ORION_WORKER_ID env var
    queue_names    TEXT[],                     -- PostgreSQL native array type
    concurrency    INT NOT NULL DEFAULT 1,
    status         VARCHAR(50) NOT NULL DEFAULT 'idle',
    last_heartbeat TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    registered_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

Workers do an UPSERT (INSERT ... ON CONFLICT DO UPDATE) on startup:
```sql
INSERT INTO workers (id, hostname, ...)
VALUES (...)
ON CONFLICT (id) DO UPDATE SET
    last_heartbeat = NOW(),
    status = 'idle';
```

Then every 15 seconds: `UPDATE workers SET last_heartbeat = NOW() WHERE id = $1`

The scheduler uses this to answer: "which workers are alive?" — any worker with `last_heartbeat > NOW() - 90 seconds`.

## Tables 4 & 5: `pipelines` and `pipeline_jobs`

For Phase 8 (DAG pipelines). `dag_spec` stores the entire pipeline graph as JSONB:

```json
{
  "nodes": [
    {"id": "preprocess", "job_template": {...}},
    {"id": "train",      "job_template": {...}},
    {"id": "evaluate",   "job_template": {...}}
  ],
  "edges": [
    {"source": "preprocess", "target": "train"},
    {"source": "train",      "target": "evaluate"}
  ]
}
```

`pipeline_jobs` maps each node to an actual `jobs` row once the jobs are created.

## The `updated_at` trigger

```sql
CREATE OR REPLACE FUNCTION update_updated_at()
RETURNS TRIGGER AS $$
BEGIN
    NEW.updated_at = NOW();
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_jobs_updated_at
    BEFORE UPDATE ON jobs
    FOR EACH ROW EXECUTE FUNCTION update_updated_at();
```

This PostgreSQL trigger fires automatically on every UPDATE. You never need to set `updated_at` in Go code — the database handles it.

## How to apply this migration

```bash
# Install the migrate tool (one time)
go install -tags 'postgres' github.com/golang-migrate/migrate/v4/cmd/migrate@latest

# Apply all pending migrations
make migrate-up

# To verify tables were created:
docker compose exec postgres psql -U orion -d orion -c "\dt"
# Should show: jobs, job_executions, pipeline_jobs, pipelines, workers

# To check indexes:
docker compose exec postgres psql -U orion -d orion -c "\di"
```
