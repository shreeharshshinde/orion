# Phase 7 Implementation Record
## gRPC Streaming API: Real-Time Job and Pipeline Events

> **What this document is:** A complete record of what was built in Phase 7, why each decision was made, and how to verify the implementation. Written after the fact as a reference for future phases and code reviewers.

---

## What Phase 7 Added

Before Phase 7, clients had to poll `GET /jobs/{id}` to track job progress. After Phase 7:

```
HTTP :8080 — all existing routes unchanged
gRPC :9090 — JobService with 4 RPCs:
  SubmitJob(SubmitJobRequest)         → Job              (unary)
  GetJob(GetJobRequest)               → Job              (unary)
  WatchJob(WatchJobRequest)           → stream JobEvent  (server-streaming)
  WatchPipeline(WatchPipelineRequest) → stream PipelineEvent (server-streaming)
```

The gRPC server runs in the same `cmd/api` binary alongside the HTTP server. Two listeners, one process, one store.

## Key Decisions

### 1. buf over protoc for code generation

The system `protoc` (v3.19.6) does not bundle well-known types (`google/protobuf/timestamp.proto`) in `/usr/include`. Rather than installing additional system packages or managing include paths manually, we use `buf` with the `buf.build/protocolbuffers/wellknowntypes` remote dependency. This is reproducible across machines with no system-level setup beyond `go install github.com/bufbuild/buf/cmd/buf@latest`.

### 2. grpc v1.64.0 required by generated code

The buf remote plugin (`buf.build/grpc/go`) generates code using `grpc.ServerStreamingServer[T]` (generic interface) and `grpc.SupportPackageIsVersion9`, which require grpc ≥ v1.64.0. The previous version in go.mod was v1.62.1. Upgraded to v1.64.0.

### 3. InstrumentedStore wrapper (not store injection)

The store package must not import the gRPC package. The `InstrumentedStore` wrapper in `internal/api/grpc/` satisfies `store.Store` and adds broadcast calls after successful DB updates. The store package stays clean. See `INSTRUMENTATION_WIRING.md` for the full rationale.

### 4. Polling + broadcaster (not PG NOTIFY yet)

Phase 7 uses a 500ms polling ticker as the primary event source, with the broadcaster as the fast path. This is simpler to implement and test without a running database. Phase 8 will replace the polling loop with `pg_notify` for sub-10ms latency. The broadcaster remains in Phase 8 — only the event source changes.

### 5. bufconn for tests (not real network)

`google.golang.org/grpc/test/bufconn` provides an in-memory connection that behaves exactly like a real TCP connection. Tests run without binding ports, work in parallel, and have zero network overhead. The test server is created fresh per test with `t.Cleanup` for teardown.

## End-to-End Verification (requires running infrastructure)

```bash
# Start infrastructure
make infra-up && make migrate-up

# Start services
make run-api       # HTTP :8080 + gRPC :9090
make run-scheduler
make run-worker

# Verify gRPC server is up
grpcurl -plaintext localhost:9090 list
# → orion.v1.JobService

# Submit a job via gRPC
JOB_ID=$(grpcurl -plaintext \
  -d '{"name":"grpc-test","type":"inline","payload":{"handler_name":"noop"}}' \
  localhost:9090 orion.v1.JobService/SubmitJob | jq -r .jobId)

# Watch it (streams until completed)
grpcurl -plaintext \
  -d "{\"job_id\":\"$JOB_ID\"}" \
  localhost:9090 orion.v1.JobService/WatchJob
```

## Architecture After Phase 7

```
cmd/api/main.go
├── pgStore (postgres.DB)
├── broadcaster (Broadcaster)
├── instrumentedStore (InstrumentedStore wraps pgStore)
│
├── HTTP :8080
│   ├── JobHandler    → uses pgStore
│   └── PipelineHandler → uses pgStore
│
└── gRPC :9090
    └── JobService    → uses instrumentedStore + broadcaster
        ├── SubmitJob  (unary)
        ├── GetJob     (unary)
        ├── WatchJob   (server-streaming)
        └── WatchPipeline (server-streaming)
```

State transition path:
```
Worker → instrumentedStore.MarkJobCompleted
       → db.MarkJobCompleted (SQL)
       → broadcaster.Publish
       → WatchJob goroutine receives event
       → stream.Send to client
```

## What Phase 8 Will Change

- Replace 500ms polling in `WatchJob` with PostgreSQL `LISTEN/NOTIFY`
- Wire the worker pool to use `instrumentedStore` (currently it uses `pgStore` directly)
- Add per-queue concurrency limits
- Add fair scheduling within priority tiers

The broadcaster, `InstrumentedStore`, and gRPC server structure remain unchanged in Phase 8.
